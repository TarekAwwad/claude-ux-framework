# Findings: existing skills review

Date: 2026-07-16. Produced by a web-research subagent; sources cited inline.

Method: fetched raw `SKILL.md` files (and supporting reference files where relevant) directly from GitHub via `raw.githubusercontent.com` and the GitHub contents API, so line counts and directory structures below are read from the actual files, not from marketing copy or search snippets.

Scope reviewed: the full skill list in `anthropics/skills` (17 skill directories, of which 6 are UI/UX/design-adjacent), the skill-authoring conventions in `obra/superpowers` (12 skills, none of which are UX-specific, so the review focuses on its `writing-skills` meta-skill and one concrete enforcement example), and 6 community UX/UI skills found via search. 15 individual `SKILL.md` files were read in full, plus one reference file (`anthropic-best-practices.md`) and one collection README.

## 1. Anthropic official skills (github.com/anthropics/skills)

Full directory listing (via GitHub API): `algorithmic-art`, `brand-guidelines`, `canvas-design`, `claude-api`, `doc-coauthoring`, `docx`, `frontend-design`, `internal-comms`, `mcp-builder`, `pdf`, `pptx`, `skill-creator`, `slack-gif-creator`, `theme-factory`, `web-artifacts-builder`, `webapp-testing`, `xlsx`. Of these, `frontend-design`, `web-artifacts-builder`, `webapp-testing`, `theme-factory`, `brand-guidelines`, and `canvas-design` are UI/UX/design-adjacent; the rest (document generation, comms, MCP tooling, GIFs) are out of scope and skipped here.

### frontend-design
- URL: https://github.com/anthropics/skills/blob/main/skills/frontend-design/SKILL.md
- Covers: aesthetic direction for building or reshaping a UI, told through a "design lead at a small studio" framing. Names three specific AI-generated design clusters to avoid (warm cream + serif + terracotta; near-black + acid accent; broadsheet/newspaper layout) and says to follow the brief exactly where it specifies one of these, but not to default into them otherwise.
- Structure: single file, 55 lines, no sub-resources beyond a license file. Not progressive disclosure; everything is inline prose.
- Enforcement: no hard gates or required artifacts. It prescribes a two-pass process (brainstorm a compact token system covering color, type, layout, and a "signature" element, then critique that plan against the brief before writing code) but nothing forces the agent through the steps; it's persuasive prose, not a checklist or gate.
- Content ideas worth adopting: (1) naming concrete, current "generic AI design" patterns by description so the agent can recognize and avoid them, rather than saying "avoid generic design"; (2) the brainstorm-then-critique-before-code two-pass structure; (3) the "spend your boldness in one place" restraint principle paired with a quality floor (responsive, focus states, reduced motion) that doesn't need announcing; (4) a short section on copywriting as design material (active voice, name things by what the user controls, treat empty/error states as direction not mood).
- To avoid: the whole skill is soft guidance with no verification step, so an agent under pressure (time, ambiguity) has nothing forcing it to actually do the plan-then-critique pass instead of skipping straight to code.

### web-artifacts-builder
- URL: https://github.com/anthropics/skills/blob/main/skills/web-artifacts-builder/SKILL.md
- Covers: building complex React/TypeScript/Tailwind/shadcn-ui artifacts for claude.ai, via an init script, edit, then a bundling script that inlines everything into one HTML file.
- Structure: single file, 73 lines, plus a `scripts/` directory with `init-artifact.sh` and `bundle-artifact.sh`. Low-freedom, script-driven workflow (run exactly these commands in this order).
- Enforcement: procedural, not a content gate. The scripts do the enforcing (they fail if `index.html` is missing, etc.) rather than the prose. One explicit content rule: "avoid excessive centered layouts, purple gradients, uniform rounded corners, and Inter font" to dodge "AI slop," stated once, not elaborated.
- Content ideas: bundling a real init/build script instead of describing the stack in prose saves tokens and removes an entire class of setup mistakes; testing is explicitly deferred ("avoid testing the artifact upfront as it adds latency... test later if requested") which is a useful sequencing lesson for a design skill (don't gate delivery on verification that isn't the point of the task).
- To avoid: the anti-"AI slop" guidance is a single unexplained line with no reasoning or examples, which is thin compared to how other skills (see addyosmani below) handle the same problem with a table of specific patterns and why each is bad.

### webapp-testing
- URL: https://github.com/anthropics/skills/blob/main/skills/webapp-testing/SKILL.md
- Covers: driving and testing local web apps with Playwright, for verifying UI behavior, screenshots, and console logs.
- Structure: single file, 95 lines, plus `scripts/with_server.py` and an `examples/` directory (`element_discovery.py`, `static_html_automation.py`, `console_logging.py`). Uses an explicit decision tree (static HTML vs. dynamic app vs. server already running) rendered as ASCII.
- Enforcement: procedural pattern, not a content gate. States one concrete "Common Pitfall": don't inspect the DOM before `networkidle` on dynamic apps. Also instructs the agent to treat bundled scripts as black boxes, run `--help` first, and not read the source unless it fails, which is a deliberate token-budget discipline (keep large scripts out of context until proven necessary).
- Content ideas: the "reconnaissance-then-action" pattern (screenshot/inspect DOM first, then act on discovered selectors) is a clean, reusable technique; the decision-tree-for-routing style is a good template for "which approach applies to this UI situation."
- To avoid: nothing to flag; it's a lean, focused, purely procedural skill with no UX/visual-design content (it's build/verify tooling, not design judgment), so it's a partial match at best for a UX skill's purposes.

### theme-factory
- URL: https://github.com/anthropics/skills/blob/main/skills/theme-factory/SKILL.md
- Covers: applying one of 10 preset color/font themes to a slide deck or artifact, or generating a new one on request.
- Structure: single file, 59 lines, plus a `themes/` directory and a `theme-showcase.pdf`. A numbered 4-step workflow (show showcase, ask, wait for explicit confirmation, apply).
- Enforcement: one real gate: "wait for explicit confirmation about the chosen theme" before applying it. That's the only step framed as a hard requirement; everything else is just numbered instructions.
- Content ideas: the "show visual choices, then require explicit user confirmation before proceeding" pattern is worth reusing anywhere a UX skill makes an aesthetic decision on the user's behalf.
- To avoid: thin content relative to a real UX skill: it's a lookup table of 10 named palettes plus font pairs, not design reasoning. Mostly relevant as a UI pattern reference (confirm-before-apply), not as design content.

### brand-guidelines
- URL: https://github.com/anthropics/skills/blob/main/skills/brand-guidelines/SKILL.md
- Covers: applying Anthropic's own brand hex codes and font pairings (Poppins/Lora with Arial/Georgia fallbacks) to artifacts.
- Structure: single file, 73 lines, no sub-resources. Pure data reference (colors, fonts, fallback rules) with no process at all.
- Enforcement: none. It's read-only reference data, not a skill with any behavior-shaping mechanism.
- Content ideas: minimal, but the fallback-font pattern (specify a webfont plus a system-font fallback with a stated reason: "no font installation required") is a small, concrete, reusable idea for any skill that touches typography.
- To avoid: this is the thinnest of the Anthropic skills reviewed; it demonstrates that a skill can just be static reference data, which is a fine pattern for narrow lookup tasks but not a model for anything that needs to change agent behavior.

### canvas-design (design-adjacent, not UI/UX)
- URL: https://github.com/anthropics/skills/blob/main/skills/canvas-design/SKILL.md
- Covers: generating static visual art (.png/.pdf posters), not interactive UI. Included here only because it surfaced in the design category; it is not a UX skill.
- Structure: single file, 129 lines, plus a large `canvas-fonts/` directory.
- Enforcement: repeated emphasis ("REPEATEDLY," "CRITICAL") that the output must look "meticulously crafted" and museum-quality, plus a scripted final self-critique pass ("refine, don't add more graphics"). This is emphasis-through-repetition rather than a structural gate.
- Content ideas: mostly not transferable to interactive UX work (it's about static art direction, "design philosophy" manifestos, and a "subtle reference" embedded in the piece).
- To avoid: heavy use of all-caps and repeated superlatives ("CRITICAL," "master-level," "countless hours") to induce quality, which reads as trying to compensate for weak structural enforcement with volume of emphasis. Worth noting as an anti-pattern: caps-lock urgency is not the same as a real gate.

### skill-creator (meta: Anthropic's own skill-building process)
- URL: https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md
- Covers: not a UX skill itself, but Anthropic's own tool and process for creating and iterating on skills (interview the user, draft `SKILL.md`, write test prompts, run with-skill vs. baseline subagents, grade against assertions, view results in a generated HTML viewer, optimize the trigger description).
- Structure: 485 lines, plus `agents/`, `assets/`, `eval-viewer/`, `references/`, and `scripts/` directories. This is itself the reference implementation of "progressive disclosure": a description-writing section, a full skill-writing guide with the canonical anatomy (`SKILL.md` + `scripts/` + `references/` + `assets/`), and pointers out to `references/schemas.md` and `agents/grader.md` for detail instead of inlining it.
- Enforcement: describes (but doesn't itself run) an eval loop: spawn with-skill and baseline subagents in the same turn, grade outputs against assertions with the exact field names the viewer expects, aggregate into `benchmark.json`, and iterate until the user is satisfied or feedback stops changing. Explicitly forbids skipping straight to editing without testing and tells the agent to add checklist items to its own todo list so steps aren't skipped.
- Content ideas directly reusable for building any new skill, including a UX one: (1) the canonical anatomy (SKILL.md required, `scripts/` for deterministic/repetitive tasks, `references/` for docs loaded on demand, `assets/` for output templates); (2) the stated word-count targets (metadata ~100 words always-in-context, body under 500 lines ideal, unlimited in bundled resources); (3) "explain to the model why things are important in lieu of heavy-handed musty MUSTs" as an explicit style directive; (4) description-writing guidance to make triggering "a little bit pushy" with concrete trigger phrases rather than vague wording.
- To avoid: this document is itself long (485 lines) and process-heavy (eval harness, grading JSON schemas, viewer scripts) in a way that's appropriate for a meta skill about skill-building, but would be excessive weight for an actual UX-content skill; it's a model for the *creation process*, not for the shape of the finished UX skill.

## 2. Superpowers plugin conventions (github.com/obra/superpowers)

The superpowers skill set (`brainstorming`, `dispatching-parallel-agents`, `executing-plans`, `finishing-a-development-branch`, `receiving-code-review`, `requesting-code-review`, `subagent-driven-development`, `systematic-debugging`, `test-driven-development`, `using-git-worktrees`, `using-superpowers`, `verification-before-completion`, `writing-plans`, `writing-skills`) is entirely a software-development-process framework. There is no UI/UX skill in this repo; the value here is purely in the meta-conventions for how a skill should be structured to actually change agent behavior, which the collection is unusually explicit and opinionated about.

### writing-skills
- URL: https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md
- Covers: how to write and edit skills, framed explicitly as "TDD applied to process documentation": write a pressure-scenario test, watch an agent fail it without the skill (baseline), write the skill, watch the agent pass, then refactor to close new rationalizations found in re-testing.
- Structure: 689 lines, single file (with sibling reference files in the same directory, e.g. `testing-skills-with-subagents.md`, `persuasion-principles.md`, `graphviz-conventions.dot`, referenced but not inlined). Includes a prescribed `SKILL.md` template (Overview, When to Use, Core Pattern, Quick Reference, Implementation, Common Mistakes, optional Real-World Impact).
- Enforcement mechanisms, several concrete and unusual ones:
  - The "Iron Law": `NO SKILL WITHOUT A FAILING TEST FIRST`, applied to new skills and edits alike, with an explicit "no exceptions" list closing off rationalizations like "just adding a section."
  - A "Skill Discovery Optimization" (SDO) rule, empirically justified with a specific before/after example: descriptions must state *only* triggering conditions, never summarize the skill's workflow, because a workflow-summarizing description caused a tested agent to follow the description's shortcut instead of reading the full flowchart (skipped a required second code-review pass).
  - A "Match the Form to the Failure" table: prohibition + rationalization table + red flags for rule-skipping-under-pressure failures; positive recipe/contract for wrong-shaped-output failures; structural required-field/slot for omitted-element failures; conditional-on-observable-predicate for context-dependent behavior. It explicitly warns that prohibitions backfire on shaping problems (cites a head-to-head wording test where a prohibition arm did worse than no guidance at all) and that "don't X unless it matters" nuance clauses reopen the exact negotiation the rule was meant to close.
  - Bulletproofing techniques: close every loophole by name ("don't keep it as reference," "don't look at it," "delete means delete"), a rationalization table built from actual observed excuses, a "Red Flags - STOP" list, and a required "no exceptions" clause on every hard rule.
  - Token-efficiency targets stated as numbers: getting-started/always-loaded content under ~150 to 200 words, other skills under ~500 words, with concrete before/after compression techniques (move flag docs to `--help`, cross-reference instead of repeating, compress examples).
- Content ideas: (1) the description-is-only-a-trigger rule, with the same reasoning, is directly applicable and testable for a UX skill's own description; (2) the "Match the Form to the Failure" table is the single most useful structural idea found in the whole review, since it gives a decision procedure for choosing checklist vs. prohibition vs. structural-required-field depending on what kind of mistake the skill is trying to prevent (a UX skill mostly needs the "wrong shape" and "omitted element" forms, not blanket prohibitions); (3) a rationalization table and red-flags list, scoped to the actual excuses an agent gives for skipping design-quality steps (e.g., "the design isn't final so I'll skip styling," seen almost verbatim in addyosmani's skill below); (4) explicit word-count and line-count budgets rather than a vague "keep it short."
- To avoid: the full TDD-for-skills loop (pressure scenarios, baseline runs, RED/GREEN/REFACTOR, rationalization-table iteration) is a heavyweight authoring *process*, not something to put inside the shipped skill itself; a UX skill should absorb the output of this process (a tight, tested rule set) without importing the whole apparatus as user-facing content.

### anthropic-best-practices.md (reference file bundled inside writing-skills)
- URL: https://github.com/obra/superpowers/blob/main/skills/writing-skills/anthropic-best-practices.md
- Covers: this is effectively Anthropic's own official skill-authoring guidance, mirrored as a reference file (headings and content match Anthropic's public "Skill authoring best practices" doc). 1150 lines, clearly meant to be read selectively rather than end to end.
- Structure and enforcement: not a skill itself, a bundled reference. Its "degrees of freedom" framework is the most reusable idea: match specificity to task fragility, ranging from high-freedom text instructions (multiple valid approaches, decisions depend on context) through medium-freedom parameterized scripts to low-freedom "run exactly this command, do not modify it" for fragile, must-not-vary operations. Also documents a "solve, don't punt" rule for any bundled scripts (handle errors in the script rather than surfacing a raw exception to the agent), a ban on unexplained magic numbers/"voodoo constants" (every constant needs a one-line reason in a comment), and a closing checklist for effective skills.
- Content ideas: the degrees-of-freedom framework maps well onto a UX skill: layout/copy/color decisions are high-freedom (many valid answers, judgment-dependent), a required design-token structure is medium-freedom (a pattern to follow with parameters), and something like a specific WCAG contrast-ratio check is low-freedom (a fixed, non-negotiable value).
- To avoid: as a reference file it duplicates a lot of general software-engineering advice (path conventions, script error handling) that isn't UX-specific; useful for the meta-question of "how do I author a good skill" but not itself UX content.

### verification-before-completion (concrete example of a hard gate in practice)
- URL: https://github.com/obra/superpowers/blob/main/skills/verification-before-completion/SKILL.md
- Covers: not UX, but chosen because it's the clearest worked example of the enforcement patterns writing-skills describes in the abstract, applied to "don't claim work is done without evidence."
- Structure: 139 lines, single file, extremely dense with tables.
- Enforcement: a genuinely hard gate. States an "Iron Law" (`NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE`), then a "Gate Function" as literal pseudocode (identify the command that proves the claim, run it fresh, read full output, only then make the claim; "skip any step = lying, not verifying"). Backs it with a "Common Failures" table mapping each type of claim to what evidence is required and what is explicitly not sufficient, a "Red Flags - STOP" list of specific phrasings that signal a violation is about to happen ("should," "probably," "Great!," "Perfect!," "Done!"), and a rationalization table pairing each excuse with a one-line rebuttal.
- Content ideas: this is the strongest template in the whole review for turning "be careful/be thorough" into something enforceable. For a UX skill, the same shape could gate claims like "this is accessible" or "this matches the design system" behind a named, concrete check (contrast ratio actually computed, keyboard nav actually tried) rather than accepting the agent's unverified assertion.
- To avoid: nothing to avoid here structurally; the only caution is that this level of "Iron Law" / all-caps intensity is calibrated for a discipline-enforcement skill guarding against a serious failure mode. Applying the same intensity to softer, more judgment-based UX guidance (like color choice) would likely backfire per writing-skills' own "Match the Form to the Failure" table, since prohibitions and iron laws are prescribed for rule-skipping-under-pressure failures, not for shaping subjective output quality.

## 3. Community skill collections

### vercel-labs/agent-skills - web-design-guidelines
- URL: https://github.com/vercel-labs/agent-skills/blob/main/skills/web-design-guidelines/SKILL.md
- Covers: reviewing UI code against Vercel's public "Web Interface Guidelines," triggered by requests like "review my UI" or "check accessibility."
- Structure: single file, only 39 lines, and unusually thin by design: it doesn't embed any actual rules. Instead it instructs the agent to `WebFetch` the live guidelines from `raw.githubusercontent.com/vercel-labs/web-interface-guidelines/main/command.md` at review time, then apply whatever that file currently contains.
- Enforcement: procedural, not content-based. It specifies the exact output format ("terse `file:line` format") but defers all substantive rules to the fetched document, so the skill itself never goes stale even as the underlying guideline set changes.
- Content ideas: the live-fetch pattern is the standout idea here: for review/audit skills, pointing at an external, versioned source of truth (fetched fresh each run) avoids the maintenance burden and staleness risk of hardcoding rules in the skill, and keeps the skill file itself tiny.
- To avoid: this pattern adds a runtime dependency (network access, the source staying up) and cannot be evaluated on its own; the actual rule content lives outside the repo entirely, so reviewing this skill in isolation tells you almost nothing about design quality, only about the fetch-then-apply mechanism.

### addyosmani/agent-skills - frontend-ui-engineering
- URL: https://github.com/addyosmani/agent-skills/blob/main/skills/frontend-ui-engineering/SKILL.md
- Covers: building production-quality, accessible, responsive UI: component architecture and file colocation, state-management selection, design-system adherence, WCAG 2.1 AA accessibility (keyboard nav, ARIA, focus management, meaningful empty/error states), responsive breakpoints, loading/transition patterns.
- Structure: single file, 328 lines, code-heavy (React/TSX examples throughout), pointing out to `references/accessibility-checklist.md` for deeper detail rather than inlining it.
- Enforcement: several concrete mechanisms. An "AI Default" table lists eight specific generic patterns (purple/indigo everything, excessive gradients, `rounded-2xl` everywhere, generic hero sections, lorem-ipsum copy, oversized padding, stock card grids, shadow-heavy design), each paired with *why* it's a problem and what production quality looks like instead, not just "avoid X." A "Common Rationalizations" table anticipates specific excuses ("accessibility is a nice-to-have," "we'll make it responsive later," "this is just a prototype") with rebuttals. A "Red Flags" list (components over 200 lines, inline styles, missing error/loading/empty states, color as sole state indicator). A closing checkbox "Verification" section the agent runs through after building (renders without console errors, tab-navigable, screen-reader conveys structure, responsive at four named breakpoints, all three states handled, follows design-system tokens, no accessibility warnings).
- Content ideas: (1) the AI-default table format (pattern, why it's a problem, what to do instead) is more actionable than a bare "avoid gradients" rule and is a pattern worth standardizing on; (2) the rationalization table targeting design-specific excuses rather than generic ones is directly reusable; (3) the closing verification checklist, phrased as checkable actions rather than vague quality goals, is a good template for a "before you call this done" gate.
- To avoid: nothing structurally weak here; it's one of the more disciplined community skills reviewed. The only limitation is scope: it's React/Tailwind-specific throughout, so it doesn't generalize to other stacks without adaptation.

### joshuadavidthomas/agent-skills - frontend-design-principles
- URL: https://github.com/joshuadavidthomas/agent-skills/blob/main/frontend-design-principles/SKILL.md
- Covers: building or reviewing frontend UI (dashboards, admin panels, landing pages, marketing sites), routing to a context-specific guide depending on whether the surface is a product/data UI or a public/creative page.
- Structure: 117-line router file, which then points to `app.md` (product/data UIs) and `marketing.md` (public/creative pages), plus a deeper `references/principles.md` for concrete spacing/depth/typography/color values. This is a clean two-level progressive-disclosure structure: routing logic and hard gates in the main file, domain content in the routed files, implementation detail one level further down.
- Enforcement: the strongest gate mechanism found in the whole community search. A "Required: Before Generating" section states plainly "Do not write code until these are done," then requires the agent to answer three intent questions (who is this human, what must they accomplish, what should this feel like, each with "do not guess, do not default"), produce four specific outputs (Domain concepts, Color world, Signature element, three named Defaults to reject), and propose a direction back to the user in a fixed template, explicitly waiting for confirmation before generating code. A parallel "Required: Before Showing" section runs four self-tests after the fact: a swap test, a squint test, a signature test, and a token test. It also states a falsifiable design principle: "if another AI given a similar prompt would produce substantially the same output, you failed."
- Content ideas: (1) the before-generating gate (answer specific questions, produce named artifacts, get confirmation) is the single best mechanism found for preventing generic AI-default output, because it forces concrete commitments before code exists rather than critiquing after the fact; (2) the before-showing self-test suite gives the agent an actual falsifiable check instead of "does this look good"; (3) the domain-routing split (product/data UI vs. marketing/creative) is a sensible top-level fork for a general UX skill, since the two contexts genuinely need different guidance; (4) "every choice must be a choice, stated with a reason" as a running principle, paired with a concrete anti-pattern list (dramatic drop shadows, arbitrary asymmetric padding, thick decorative borders, multiple accent colors, mixed depth strategies, inconsistent spacing).
- To avoid: it explicitly punts spacing/depth/typography/color implementation detail to `references/principles.md` (not fetched in this review), so on its own the main file is more about *process* (ask, propose, confirm, self-test) than about concrete visual rules; a UX skill following this pattern needs to make sure the routed-to reference files are actually as well-built as the router.

### tommyjepsen/awesome-ux-skills (collection, 20 separate skills) and craft.md
- URLs: https://github.com/tommyjepsen/awesome-ux-skills and https://github.com/tommyjepsen/awesome-ux-skills/blob/main/craft.md
- Covers: a flat collection of 20 single-file skills grouped into three categories: UX Research & Strategy (`ux-research-methods`, `ux-personas`, `empathy-mapping`, `journey-mapping`, `ux-storyboard`, `double-diamond`, `like-wish-what-if`), UI Analysis & Improvement (`general-design-review`, `accessibility`, `ux-heuristics-review`, `craft`, `cognitive-load-conversion`, `persuasive-ux`, `feature-prioritization`), and AI Product Design (`ai-governors`, `ai-identifiers`, `ai-inputs`, `ai-trust-builders`, `ai-tuners`, `ai-wayfinders`).
- Structure: unusually, this repo is a flat directory of `.md` files at the repo root rather than one folder per skill; an `install.sh` script presumably copies each into `~/.claude/skills/<name>/SKILL.md` at install time.
- Craft.md specifically (83 lines) is the most concrete example: it defines exactly 12 named, numbered rules (R1 through R12: no gradients, no glow, no `transition: all`, kill visual monotony, no placeholder text, `isolation: isolate` for stacking contexts, no pure black/white, spacing on a scale, a small type scale, pick one elevation language, design every interactive/loading/empty/error state, motion as physics not decoration). Each rule states the problem, the exception if any, and the concrete fix (real CSS).
- Enforcement: craft.md has two explicit modes (critique: check code against the 12 rules and report only violations with concrete fixes; build: apply all 12 silently, mention only the ones that shaped a visible decision) and a fixed output contract for critique mode (one-line verdict using a fixed three-way emoji scale, then only the violated rules each with a fix, capped at "Priority Fixes: max 3, ordered by visual impact"). It explicitly forbids padding the list.
- Content ideas: (1) numbering rules (R1, R2...) makes them citable in critique output; (2) the two-mode split (critique vs. build) with different behavior for each is a clean pattern most of the other skills in this review don't have; (3) capping "priority fixes" at a small fixed number, and explicitly forbidding invented violations, keeps critique output honest and useful rather than exhaustive and overwhelming; (4) the AI Product Design category (`ai-governors`, `ai-trust-builders`, `ai-wayfinders`, etc.) is a genuinely underserved niche among the skills reviewed, worth borrowing topic coverage from even where the execution is thin.
- To avoid: the collection as a whole is dozens of thin, independent single-file skills, several of which likely overlap heavily in triggering conditions (`general-design-review`, `ux-heuristics-review`, and `craft` could all plausibly fire on the same "review my UI" request), and the repo provides no visible evidence of testing per superpowers' RED/GREEN/REFACTOR standard; craft.md itself is the strongest file in the set, the others were not individually verified in this pass beyond the README's descriptions.

### NickCrew/claude-cortex - dashboard-designer
- URL: https://github.com/NickCrew/claude-cortex/blob/main/skills/dashboard-designer/SKILL.md
- Covers: dashboard design end to end: defining audience and KPIs, information architecture (F-pattern, Z-pattern, grid layouts with ASCII diagrams), visual hierarchy, a color-coding system, a BI-tool selection guide, a quality checklist, and two worked examples.
- Structure: single file, 249 lines, no sub-resources.
- Enforcement: none beyond an eight-item "Dashboard Quality Checklist" near the end, presented as a checklist to "verify" before publishing but with nothing structurally forcing the agent to run it.
- Content ideas: (1) the audience-first framing ("who uses this, what decision does it inform, how often will they look at it, what action can they take, if no action then reconsider whether a dashboard is the right format") is a genuinely good opening gate for any dashboard-building skill, forcing purpose before layout; (2) the KPI-tiering scheme (Level 1 headline / Level 2 supporting / Level 3 detail, with stated counts per audience type) is concrete and directly usable; (3) titling charts as insights ("Revenue Up 12%") rather than labels ("Revenue") is a small, specific, high-value rule.
- To avoid: this reads as largely generated/compiled reference content rather than a tested behavior-shaping skill: it uses decorative emoji as a "color legend" inside what's meant to be a professional design document, which itself resembles the generic-AI-output pattern that other skills in this review explicitly warn against. It's a solid knowledge dump but has no gate, no rationalization table, and no verification step forcing the checklist to actually run.

### szilu/ux-designer-skill
- URL: https://github.com/szilu/ux-designer-skill/blob/main/SKILL.md
- Covers: an extremely broad UX reference spanning core principles, Laws of UX, WCAG 2.2 accessibility, visual design, information architecture, interaction design, forms, mobile UX, UX writing, user research, design systems, real-time collaboration/presence, canvas/whiteboard UX, AI interface patterns, ethical design, onboarding, notifications, data visualization, search UX, emotional design, data tables, performance UX, internationalization, and voice/multimodal UX.
- Structure: a 325-line router `SKILL.md` (overview, checklists per topic area, two ASCII decision trees, a "Detailed Documentation" index, a "Key Numbers to Remember" set of tables, a 25-item numbered "Anti-Patterns to Avoid" list each linking to a specific reference file) sitting on top of 26 numbered reference files (`01-core-principles.md` through `24-voice-and-multimodal.md`, with `12a/12b` and `13a/13b` splits) totaling roughly 345 KB of additional content, none of which was fetched in this pass beyond file names and sizes.
- Enforcement: none structural. It's checklists throughout (flat lists of unchecked boxes) rather than gates; nothing requires the agent to actually complete a checklist or produce a verifiable artifact before proceeding.
- Content ideas: (1) the "Key Numbers to Remember" table (touch target 44 to 48px, contrast ratio 4.5:1, animation 300 to 500ms, text expansion ~30 to 40% for translation, and similar) is a genuinely useful compact reference of hard-to-recall constants worth a similar table in any UX skill; (2) linking each anti-pattern to the specific reference file that explains it is a clean way to keep the router thin while still being a real index; (3) the "Sources" section citing Laws of UX, NN/g, WCAG, Material Design, Apple HIG, Baymard, ColorBrewer, and others by name signals where the content actually comes from, which is a good transparency habit even for a router file.
- To avoid: this is the clearest example in the review of a large, comprehensive knowledge dump duplicating a great deal of general UX knowledge a capable model plausibly already has, with no mechanism that changes agent behavior under pressure, no tested rationalization handling, and no verification gate. At ~350KB of reference content it also blows well past every "keep it lean" budget cited elsewhere in this review (Anthropic's own <500-line-ideal, superpowers' <500-word target for most skills).

## Recommendations for ux-framework

Ranked by how much leverage each idea plausibly gives a new UX skill, drawing on the strongest instance of each pattern across all three sources.

1. **Separate the trigger from the workflow.** The description field should state only when to use the skill, never summarize what the skill does step by step, because superpowers demonstrated with a real before/after test that a workflow-summarizing description causes agents to substitute the summary for actually reading the skill and skip required steps. (https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md)

2. **Gate generation on a small set of concrete, named commitments, not vibes.** The strongest anti-genericness mechanism found anywhere in this review is joshuadavidthomas's "Required: Before Generating" gate: answer who/what/feel-like without guessing, produce four named outputs (domain vocabulary, a color world grounded in that domain, one true signature element, three defaults explicitly rejected), and get user confirmation before writing code. This beats prose exhortations to "be distinctive" because it produces a checkable artifact instead of a mood. (https://github.com/joshuadavidthomas/agent-skills/blob/main/frontend-design-principles/SKILL.md)

3. **Pair every gate with a falsifiable self-test, run after the fact.** The same skill's "Before Showing" suite (swap test, squint test, signature test, token test) turns "does this look good" into specific, repeatable checks. Combine this with addyosmani's closing Verification checklist for a two-sided gate: commit before, verify after. (https://github.com/joshuadavidthomas/agent-skills/blob/main/frontend-design-principles/SKILL.md, https://github.com/addyosmani/agent-skills/blob/main/skills/frontend-ui-engineering/SKILL.md)

4. **Match the enforcement form to the failure it targets, don't default to blanket prohibitions.** Superpowers' explicit finding: prohibition lists work for rule-skipping-under-pressure failures, but backfire (tested, measured) on shaping problems like generic output, where a positive recipe stating what the output IS beats a list of what not to do. For a UX skill this argues for recipes and required-field checklists over "don't use gradients" style bans, reserving hard prohibitions for genuinely binary rules (contrast ratio, focus-visible states). (https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md)

5. **Name specific, current generic-AI patterns rather than saying "avoid generic."** Anthropic's frontend-design, web-artifacts-builder, and addyosmani's frontend-ui-engineering independently converge on the same technique: list actual recognizable patterns paired with why each is a problem and what to do instead, which is more actionable than an abstract instruction. (https://github.com/anthropics/skills/blob/main/skills/frontend-design/SKILL.md, https://github.com/anthropics/skills/blob/main/skills/web-artifacts-builder/SKILL.md, https://github.com/addyosmani/agent-skills/blob/main/skills/frontend-ui-engineering/SKILL.md)

6. **Route by context, keep the router thin, push depth into reference files.** Both Anthropic's `skill-creator` guidance and joshuadavidthomas's app.md/marketing.md split point the same direction: a UX skill likely needs at least a product/data-UI branch and a marketing/creative branch. Keep the top-level file under roughly 500 lines and push concrete values into a loaded-on-demand reference file, the way szilu's 26-file reference set does, but without szilu's mistake of also inflating the router itself. (https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md, https://github.com/joshuadavidthomas/agent-skills/blob/main/frontend-design-principles/SKILL.md, https://github.com/szilu/ux-designer-skill/blob/main/SKILL.md)

7. **Build a rationalization table and red-flags list from actual excuses, not hypothetical ones.** Both verification-before-completion and addyosmani's frontend-ui-engineering pair a short list of specific excuses agents give with one-line rebuttals, and per superpowers' own testing methodology these should be built from observed failures during the skill's own test-and-iterate process. (https://github.com/obra/superpowers/blob/main/skills/verification-before-completion/SKILL.md, https://github.com/addyosmani/agent-skills/blob/main/skills/frontend-ui-engineering/SKILL.md)

8. **For audit/critique mode, use a small numbered rule set with a fixed, capped output contract.** tommyjepsen's craft.md (R1 to R12, one-line verdict, violations only, fixes capped at three by visual impact, explicit ban on inventing violations to pad the list) is a better model for a "review my UI" mode than an open-ended critique, because it produces consistent, scannable, non-overwhelming output and makes each finding citable by rule number. (https://github.com/tommyjepsen/awesome-ux-skills/blob/main/craft.md)

Two cross-cutting cautions worth carrying forward: don't let the skill become a knowledge dump duplicating things the model already knows (szilu's 350KB reference set is the cautionary example), and don't substitute repeated emphasis (all-caps, "CRITICAL," superlatives, as in canvas-design) for an actual structural gate; the skills that changed behavior most convincingly in this review did it with required artifacts, confirmation steps, and falsifiable self-tests, not with volume of insistence.
